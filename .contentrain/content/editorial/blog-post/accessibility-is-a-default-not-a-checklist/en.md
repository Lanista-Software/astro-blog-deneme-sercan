## The checklist trap

Most teams meet accessibility at the end of a project, as a list of contrast ratios and alt-text fields to fill in before launch. It gets treated like a compliance task instead of a design decision.

That order of operations guarantees friction. Retrofitting focus states, semantic structure, and keyboard navigation onto a finished interface is always more expensive than designing with them from the start.

## What changes when it's a default

When accessibility is a default assumption rather than a final pass, small things shift early: headings follow a real hierarchy because someone needs to navigate by them, color is never the only signal because someone can't rely on it, and interactive elements are built as real controls instead of styled divs.

None of this requires exotic tooling. It requires deciding, at the moment a component is designed, that it has to work for more than one kind of user.

## The payoff is broader than compliance

Interfaces built this way tend to be clearer for everyone — better keyboard support benefits power users, better contrast benefits anyone in bright light, and clean semantic structure benefits search engines as much as screen readers.

Accessibility done early isn't a tax on design. It's what disciplined design looks like.
